Section 06

Page-by-Page Walkthrough

Page 1: Overview — Market Dashboard

The Overview page serves as the landing dashboard showing the complete stock market data loaded from stock_data.h. It demonstrates how the shared data flows into all 5 modules.

  • Live Ticker Bar — Scrolling stock prices across the top (AAPL, GOOGL, MSFT...) with green/red indicators for gains/losses
  • Stock Grid — All 12 sample stocks displayed as interactive cards with symbol, price, change%, and volume
  • Module Cards — Navigation cards for each of the 5 data structures with descriptions and complexity badges
  • Server Status Badge — Shows "server.py live" when the Python server is connected
ADS Concepts on This Page

The Overview page loads all 12 StockRecord entries from the shared SAMPLE_STOCKS[] array defined in stock_data.h. This demonstrates the shared data model — the same dataset is used by the Hash Table for storage, the Priority Queue for alerts, the AVL Tree for time-series, the Heap for ranking, and the Trie for search.

Page 2: Hash Table — Stock Data Storage

This page demonstrates the Hash Table with Chaining data structure. Users can insert stock symbols and see them hashed into buckets with collision chains visualized.

  • Stat Pills — Total stocks, table size (100), load factor, max chain length, collisions count
  • Insert/Search Form — Enter stock symbol and price to insert; search by symbol to find
  • Hash Bucket Visualization — Visual grid showing which buckets are occupied, with linked-list chains for collisions
  • Stock Table — Tabular view of all inserted stocks with symbol, price, volume, hash index
  • Console — Real-time log of INSERT, COLLISION, SEARCH operations
  • C Terminal — "Run Live" executes the actual C code and shows terminal output
ADS Concepts on This Page

Hash Table — djb2 hash function maps stock symbols to bucket indices. Separate Chaining — Collisions are resolved using linked lists at each bucket. Load Factor = total_stocks / TABLE_SIZE. Key Concept Tags: DJB2 HASH, SEPARATE CHAINING, O(1) AVG LOOKUP, LOAD FACTOR, COLLISION HANDLING.

Page 3: Priority Queue — Alert Processing

This page demonstrates the Min-Heap Priority Queue. Stock alerts arrive in random order but are processed by urgency, not arrival time.

  • Stat Pills — Queue size, total inserted, total processed, next priority
  • Enqueue Alert Form — Enter symbol, price, priority level (1=CRITICAL to 4=LOW), alert type
  • Queue State — Visual list showing all alerts in heap order, color-coded by priority level
  • Action Buttons — Enqueue, Process Next (extract-min), Peek, Load Demo, Reset
  • Console — Shows ENQUEUE and PROCESS operations with timestamps
ADS Concepts on This Page

Min-Heap Priority Queue — Lower priority number = higher urgency. heapify_up() after insert, heapify_down() after extract. CRITICAL alerts (priority 1) always processed before LOW alerts (priority 4), regardless of insertion order. Key Concept Tags: MIN-HEAP ORDERING, O(LOG N) INSERT, HEAPIFY_UP ON ENQUEUE, O(1) PEEK, PRIORITY-FIRST DISPATCH.

Page 4: AVL Tree — Historical Price Data

This page demonstrates the Self-Balancing AVL Tree. Stock prices are inserted with timestamps and the tree automatically balances itself using rotations.

  • Stat Pills — Records count, tree height, rotations performed, balance factor, min/max price
  • Insert Form — Enter price and volume; timestamp is auto-generated
  • Load 20 Records — Bulk insert 20 price records to demonstrate balancing
  • Price History — Time-sorted inorder traversal showing all prices chronologically
  • Tree Visualization — SVG rendering of the AVL tree structure with nodes and edges
  • Console — Shows INSERT and ROTATION events (Left rotation, Right rotation)
ADS Concepts on This Page

AVL Tree — Self-balancing BST where |balance factor| ≤ 1 at every node. Four rotation cases: LL (Right Rotation), RR (Left Rotation), LR (Left-Right), RL (Right-Left). Inorder traversal gives time-sorted prices. Key Concept Tags: LL/RR/LR/RL ROTATIONS, SELF-BALANCING ON INSERT, O(LOG N) INSERT & SEARCH, TIME-SERIES SORTED, BALANCE FACTOR.

Page 5: Heap — Top Gainers/Losers Ranking

This page demonstrates Max-Heap and Min-Heap used simultaneously for dual-direction ranking — top gainers and top losers.

  • Stat Pills — Gainers count, losers count, best stock, worst stock
  • Add Stock Form — Enter symbol, price, change%, volume
  • Action Buttons — Add, Extract Gainer (extract-max), Extract Loser (extract-min), Demo, Reset
  • Top Gainers List — Bar chart visualization ranked by +change% (max-heap order)
  • Top Losers List — Bar chart visualization ranked by -change% (min-heap order)
  • Console — Shows INSERT, EXTRACT operations with heapify details
ADS Concepts on This Page

Dual Heap Architecture — Positive change% stocks go into the max-heap (gainers), negative change% stocks go into the min-heap (losers). This implements a Double Ended Priority Queue concept from Unit 2. Extract top-K elements from each heap to display leaderboards. Key Concept Tags: MAX-HEAP FOR GAINERS, MIN-HEAP FOR LOSERS, HEAPIFY_UP INSERT, HEAPIFY_DOWN EXTRACT, O(1) PEEK TOP.

Page 6: Trie — Stock Symbol Auto-Complete

This page demonstrates the Trie (Prefix Tree) for fast prefix-based search and auto-complete functionality.

  • Stat Pills — Symbols stored (18), trie nodes created (58), memory usage (~11.8KB)
  • Add Symbol Form — Insert new stock symbols with prices into the trie
  • Exact Search — Search for a specific symbol (AAPL, GOOGL, etc.)
  • Live Auto-Complete — Type a prefix (A, AM, NV, G...) and see real-time suggestions with highlighted matching prefix
  • All Symbols Display — Grid of all 18 symbols stored in the trie (AAPL, GOOGL, MSFT, AMZN, TSLA, META, NVDA, NFLX, AMD, INTC, ORCL, IBM, ADBE, CSCO, AVGO, TXN, QCOM, GOOG)
  • Console — Shows INSERT, SEARCH, and SUGGEST operations
ADS Concepts on This Page

Trie (Prefix Tree) — 26-ary tree where each node represents a character. Insertion follows the path character-by-character. Search is O(m) where m = key length. Auto-complete traverses to the prefix node, then collects all descendants — O(m + number of results). Space-time tradeoff: uses more memory (26 pointers per node) but achieves faster lookups than hash tables for prefix queries. Key Concept Tags: PREFIX TREE STRUCTURE, O(M) SEARCH, AUTO-COMPLETE O(M+R), SPACE-TIME TRADEOFF, 26-ARY BRANCHING.
