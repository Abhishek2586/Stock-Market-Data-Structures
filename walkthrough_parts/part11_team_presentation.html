Member 1 — Hash Table Specialist
File: member1_hashtable.c · Data Structure: Hash Table with Chaining

Presentation Script

"I implemented the Hash Table module which provides O(1) average-time stock lookup."

"My module uses the djb2 hash function — a well-known algorithm by Daniel Bernstein that multiplies the hash by 33 and adds each character. This maps stock symbols like 'AAPL' to bucket indices in a 100-element array."

"When two symbols hash to the same bucket — a collision — I use separate chaining with linked lists. Each bucket is the head of a linked list that stores all stocks at that index."

"The load factor is total_stocks / table_size. With 12 stocks in 100 buckets, it's 0.12 — very low, meaning few collisions. In the demo, I insert all 12 stocks, perform lookups, and display the chain distribution."

Key concepts I can explain: djb2 hash function, separate chaining, collision handling, load factor, O(1) amortized complexity.

Member 2 — Priority Queue Specialist
File: member2_priority_queue.c · Data Structure: Min-Heap Priority Queue

Presentation Script

"I implemented the Priority Queue module using a Min-Heap for urgent stock alert processing."

"In a stock market, CRITICAL alerts like trading halts must be processed before routine price updates. My min-heap priority queue ensures the most urgent alert (lowest priority number) is always at the root."

"When an alert is inserted, heapify_up bubbles it upward if it has higher urgency than its parent. When the top alert is extracted, heapify_down pushes the replacement element to the correct position."

"The demo inserts 10 alerts in random priority order, then extracts them all — you can see CRITICAL alerts always come out first, regardless of when they were inserted. Both insert and extract are O(log n)."

Key concepts I can explain: Min-heap property, heapify_up, heapify_down, O(log n) insert/extract, O(1) peek, parent/child index formulas.

Member 3 — AVL Tree Specialist
File: member3_avltree.c · Data Structure: AVL Tree (Self-Balancing BST)

Presentation Script

"I implemented the AVL Tree module for storing historical stock prices in a self-balancing binary search tree."

"An AVL tree maintains the balance factor — the height difference between left and right subtrees — at every node within {-1, 0, +1}. If insertion causes an imbalance, automatic rotations restore balance."

"There are four rotation cases: LL (right rotation), RR (left rotation), LR (left-right double rotation), and RL (right-left double rotation). Each rotation takes O(1) time."

"In the demo, I insert 20 price records with timestamps. The tree rebalances itself through rotations, maintaining O(log n) height. The inorder traversal shows prices sorted by timestamp."

Key concepts I can explain: Balance factor, LL/RR/LR/RL rotations, O(log n) guaranteed worst-case, height update after insertion, inorder traversal for sorted output.

Member 4 — Heap Specialist
File: member4_heap.c · Data Structure: Max-Heap & Min-Heap (Dual)

Presentation Script

"I implemented the dual Heap module — a Max-Heap for top gainers and a Min-Heap for top losers."

"Stocks with positive change go into the max-heap (gainers), where the root always holds the biggest gainer. Stocks with negative change go into the min-heap (losers), where the root holds the biggest loser."

"This effectively implements a Double Ended Priority Queue (DEPQ) — we can extract the maximum from one heap and the minimum from another in O(log n) each."

"The heapify_up function uses a boolean flag is_max_heap to decide comparison direction. The same code handles both max and min heaps with a single flag."

Key concepts I can explain: Max-heap vs min-heap, DEPQ, heapify_up/down, top-K extraction, array-based complete binary tree, parent-child index formulas.

Member 5 — Trie Specialist
File: member5_trie.c · Data Structure: Trie (Prefix Tree)

Presentation Script

"I implemented the Trie module for stock symbol auto-complete functionality."

"A trie is a 26-ary tree where each node has up to 26 children (one per letter). To insert 'AAPL', we create nodes A → A → P → L, marking L as an end-of-word."

"For auto-complete, we traverse to the prefix node (e.g., 'A'), then run a DFS to collect all complete words below it — returning AAPL, AMD, AMZN, ADBE, AVGO."

"Search is O(m) where m is the key length — independent of how many symbols are stored. This is the key advantage over hash tables for prefix queries."

"The space-time tradeoff is important: each node uses 26 pointers (even if most are NULL), but this enables instant prefix matching. With 18 symbols, we create 58 nodes using about 11.8KB."

Key concepts I can explain: 26-ary branching, O(m) search, prefix matching, DFS collection, space-time tradeoff, is_end flag.
