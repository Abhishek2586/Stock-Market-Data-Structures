Section 01

Introduction — What ADS Concepts We Use & Why

StockADS is a stock market data management system that applies 5 Advanced Data Structures to solve real-world financial data problems. Each data structure is chosen because it uniquely solves a specific stock market requirement that no other data structure can solve as efficiently.

Why These 5 ADS Concepts?

#ADS ConceptWhat It Does in Our ProjectWhy We Need It
1 Hash Table Maps stock symbols to their data (price, volume, change) using djb2 hash function with separate chaining A stock exchange needs O(1) average-time lookup — when a trader queries "AAPL", the response must be instantaneous. No other structure gives constant-time lookup by key.
2 Priority Queue Processes stock market alerts in urgency order using a min-heap — CRITICAL alerts before INFO alerts In a trading floor, a trading halt alert must be processed before a routine price update. A priority queue ensures the most urgent item is always at the top — O(log n) insert/extract.
3 AVL Tree Stores historical stock prices in a self-balancing BST keyed by timestamp, with automatic rotations Price records arrive in timestamp order, which would degenerate a regular BST into a linked list (O(n)). AVL rotations guarantee O(log n) worst-case — essential for time-series data.
4 Heap Dual max-heap (top gainers) and min-heap (top losers) for instant market ranking A stock screener needs to show the "top gainer" and "top loser" at any time. A heap gives O(1) peek at the extremum and O(log n) extraction — perfect for ranking.
5 Trie 26-ary prefix tree for stock symbol auto-complete — type "A" and instantly get AAPL, AMD, AMZN Search bars need prefix-based suggestions as the user types. A trie finds all matches in O(m + r) time — independent of how many symbols are stored. Hash tables cannot do prefix queries.
Important — What Makes This Advanced Data Structures?

These are not basic data structures like arrays or linked lists. The ADS syllabus specifically covers AVL Trees, Heaps, Priority Queues, Tries, and Hashing — all of which we implement from scratch in C. Each structure demonstrates a specific algorithmic advantage (balancing, heapification, prefix matching) that basic structures cannot provide.

Section 02

Architecture — How ADS Components Connect

Layer 1: Frontend (HTML/CSS/JS) — The User Interface

6 interactive pages where users interact with the data structures visually. The frontend sends commands and displays ADS-processed results with real-time console output.

Layer 2: Python Server (server.py) — Compilation & Execution Bridge

Compiles all C modules via gcc and executes specific modules through subprocess.run() with stdin/stdout piping. Routes HTTP requests to the appropriate C module.

Layer 3: C Backend (5 Modules) — Where All ADS Concepts Live

  • member1_hashtable.c — djb2 hash function + separate chaining for O(1) stock lookup
  • member2_priority_queue.c — Min-heap priority queue for urgency-based alert processing
  • member3_avltree.c — Self-balancing BST with LL/RR/LR/RL rotations for historical prices
  • member4_heap.c — Dual max/min heaps for gainers/losers ranking
  • member5_trie.c — 26-ary prefix tree for symbol auto-complete

How Data Flows During a Module Execution

Step 1: User clicks a tab (e.g., "01 HASH TABLE") in the frontend | Step 2: Frontend sends HTTP request to Python server (GET /run?module=1) | Step 3: server.py executes: subprocess.run("./stock_market", input="1", ...) | Step 4: main.c reads stdin choice "1" -> calls run_hashtable() | Step 5: Hash Table module executes: - Loads 12 stocks from stock_data.h - Applies djb2 hash to each symbol - Inserts into 100-bucket array with chaining - Prints formatted output to stdout | Step 6: server.py captures stdout, returns as HTTP response | Step 7: Frontend renders output in the console panel + Updates visualization (bucket display, statistics cards)
Key Architectural Point

All 5 data structures are implemented in pure C — no external libraries. The stock_data.h header provides a single source of truth: 12 StockRecord structs containing symbol, price, change_percent, and volume. Every module imports this shared data and processes it through its specific ADS algorithm.

Section 04 — Member 1

Hash Table with Chaining

What Is It? (Explain to Faculty)

A Hash Table is a data structure that maps keys to values using a hash function. The hash function converts a key (e.g., stock symbol "AAPL") into an array index. This enables O(1) average-time insert, search, and delete — the fastest possible lookup for key-value data.

When two keys map to the same index (a collision), we need a strategy. Our project uses separate chaining — each bucket stores a linked list of all elements that hash to that index.

Where Is It Applied? (Code)

The djb2 Hash Function

// File: member1_hashtable.c → hash_function()
// Daniel Bernstein's djb2 — one of the most effective string hash algorithms

unsigned int hash_function(const char* key) {
    unsigned long hash = 5381;         // Magic starting value
    int c;
    while((c = *key++))
        hash = ((hash << 5) + hash) + c;  // hash * 33 + c
    return hash % TABLE_SIZE;           // Modulo 100 → bucket index
}

Explain to faculty: "This is the djb2 hash function by Daniel Bernstein. It starts with 5381, then for each character: multiplies the hash by 33 (via bit shift + addition) and adds the character's ASCII value. The modulo operation maps the result to one of our 100 buckets. This produces a uniform distribution — minimizing collisions."

Collision Handling — Separate Chaining

// File: member1_hashtable.c → ht_insert()
// When AAPL hashes to index 11 and AMZN also hashes to 11 → COLLISION

void ht_insert(HashTable* ht, StockRecord stock) {
    int index = hash_function(stock.symbol);  // Get bucket index

    // Check if already exists — traverse the chain
    HashNode* curr = ht->buckets[index];
    while(curr) {
        if(strcmp(curr->stock.symbol, stock.symbol) == 0) {
            curr->stock = stock;  // Update existing entry
            return;
        }
        curr = curr->next;
    }

    // New entry — prepend to linked list at this bucket
    HashNode* node = malloc(sizeof(HashNode));
    node->stock = stock;
    node->next = ht->buckets[index];  // Point to old head
    ht->buckets[index] = node;          // New head
    ht->count++;
}

Explain to faculty: "When two symbols hash to the same index — like AAPL and AMZN both mapping to index 11 — we have a collision. Instead of finding another slot (open addressing), we use separate chaining: each bucket is a linked list. New collisions are prepended in O(1). Search traverses the chain — O(1) average with a good hash function and low load factor."

Syllabus Connection

Foundation: Hashing fundamentals — hash functions, collision resolution strategies. Our project implements separate chaining (linked list at each bucket). The syllabus also covers open addressing (linear/quadratic probing, double hashing) — explain the tradeoff: chaining is simpler and handles high load factors better; open addressing has better cache performance but degrades past ~70% load factor.

Section 05 — Member 2

Priority Queue — Min-Heap

What Is It? (Explain to Faculty)

A Priority Queue is an abstract data type where each element has a priority. Unlike a regular queue (FIFO), elements are dequeued in priority order — the highest-priority element comes out first, regardless of when it was inserted.

We implement it using a Min-Heap — a complete binary tree where every parent node has a priority value ≤ its children. The root always holds the minimum (most urgent) element.

Where Is It Applied? (Code)

Heapify Up — Restoring Heap After Insert

// File: member2_priority_queue.c → heapify_up()
// After inserting at the end, bubble the element UP until heap property holds

void heapify_up(PriorityQueue* pq, int idx) {
    while(idx > 0) {
        int parent = (idx - 1) / 2;          // Parent index formula
        if(pq->heap[parent].priority > pq->heap[idx].priority) {
            swap(&pq->heap[parent], &pq->heap[idx]);  // Parent is bigger → swap
            idx = parent;                               // Continue up
        } else break;                               // Heap property satisfied
    }
}

Explain to faculty: "After inserting a new alert at the end of the array, heapify_up compares it with its parent. If the new element has higher urgency (lower priority number), it swaps upward. This continues until the root is reached or the heap property is satisfied. This is O(log n) — the height of the tree."

Heapify Down — Restoring Heap After Extract

// File: member2_priority_queue.c → heapify_down()
// After extracting root and moving last element to root, push it DOWN

void heapify_down(PriorityQueue* pq, int idx) {
    int size = pq->size;
    while(1) {
        int smallest = idx;
        int left  = 2 * idx + 1;              // Left child index
        int right = 2 * idx + 2;              // Right child index

        if(left < size && pq->heap[left].priority < pq->heap[smallest].priority)
            smallest = left;
        if(right < size && pq->heap[right].priority < pq->heap[smallest].priority)
            smallest = right;

        if(smallest == idx) break;           // Heap property satisfied
        swap(&pq->heap[idx], &pq->heap[smallest]);
        idx = smallest;
    }
}

Explain to faculty: "After extracting the root (most urgent alert), we move the last element to the root position. heapify_down then pushes it to the correct position by comparing with both children and swapping with the smaller child. This maintains the min-heap invariant in O(log n)."

Section 06 — Member 3

AVL Tree — Self-Balancing BST

What Is It? (Explain to Faculty)

An AVL Tree (Adelson-Velsky and Landis, 1962) is a Binary Search Tree that automatically rebalances after every insertion or deletion. The balance factor at every node — defined as height(left subtree) - height(right subtree) — must be in {-1, 0, +1}. If an operation causes |BF| > 1, one or two rotations restore balance.

This guarantees O(log n) worst-case for all operations — unlike a plain BST which can degrade to O(n) with sorted input.

Where Is It Applied? (Code)

Rotation Cases — The Core of AVL

// File: member3_avltree.c
// RIGHT ROTATION (for LL imbalance — left-left heavy)
PriceNode* rotate_right(PriceNode* y) {
    PriceNode* x  = y->left;        // x becomes new root
    PriceNode* T2 = x->right;       // T2 = x's right subtree
    x->right = y;                   // y drops down to x's right
    y->left  = T2;                  // T2 attaches to y's left
    // Update heights
    y->height = 1 + MAX(avl_height(y->left), avl_height(y->right));
    x->height = 1 + MAX(avl_height(x->left), avl_height(x->right));
    return x;  // x is the new root of this subtree
}

// LEFT ROTATION (for RR imbalance — mirror of above)
PriceNode* rotate_left(PriceNode* x) {
    PriceNode* y  = x->right;
    PriceNode* T2 = y->left;
    y->left  = x;
    x->right = T2;
    x->height = 1 + MAX(avl_height(x->left), avl_height(x->right));
    y->height = 1 + MAX(avl_height(y->left), avl_height(y->right));
    return y;
}

AVL Insert with Auto-Rebalancing

// File: member3_avltree.c → avl_insert()
// After standard BST insert, check balance factor and rotate if needed

int balance = avl_balance(node);

// Case 1: LL — Left-Left heavy
if(balance >  1  && key < node->left->key)
    return rotate_right(node);           // Single right rotation

// Case 2: RR — Right-Right heavy
if(balance < -1  && key > node->right->key)
    return rotate_left(node);            // Single left rotation

// Case 3: LR — Left-Right (double rotation)
if(balance >  1  && key > node->left->key) {
    node->left = rotate_left(node->left);   // First: left-rotate left child
    return rotate_right(node);               // Then: right-rotate node
}

// Case 4: RL — Right-Left (double rotation)
if(balance < -1  && key < node->right->key) {
    node->right = rotate_right(node->right); // First: right-rotate right child
    return rotate_left(node);                // Then: left-rotate node
}

Explain to faculty: "After every BST insertion, we walk back up the tree updating heights. At each ancestor, we compute the balance factor. If |BF| > 1, the tree is imbalanced and we apply the corresponding rotation. Single rotations fix LL/RR cases. Double rotations fix LR/RL cases — first rotate the child, then the node. Each rotation is O(1), and we do at most O(log n) rotations per insert."

Why AVL for Stock Prices?

ScenarioPlain BSTAVL Tree
Insert 20 timestamps in orderHeight = 20 (linked list!)Height ≤ 5 (balanced)
Search for timestamp TO(20) = O(n)O(5) = O(log n)
Worst-case guaranteeNone — depends on input orderAlways O(log n)
Syllabus Connection

Unit 1: AVL Trees — Our project implements the complete AVL tree with all 4 rotation cases. The syllabus also covers Red-Black Trees (relaxed balance: height ≤ 2·log(n+1)), B-Trees (multi-way for disk), Splay Trees (amortized O(log n) via splaying). Explain: AVL is the strictest balanced BST (BF ≤ 1), making it ideal for demonstrating balancing concepts. Red-Black trades slightly worse balance for fewer rotations per insert.

Section 07 — Member 4

Heap — Max/Min Ranking

What Is It? (Explain to Faculty)

A Heap is a complete binary tree stored as an array where every parent satisfies the heap property. In a Max-Heap, every parent ≥ its children (root = maximum). In a Min-Heap, every parent ≤ its children (root = minimum).

Our project uses both simultaneously: a max-heap ranks top gainers and a min-heap ranks top losers. This combination implements a Double Ended Priority Queue (DEPQ) — extracting both the maximum and minimum efficiently.

Where Is It Applied? (Code)

Dual-Mode Heap Insert

// File: member4_heap.c → heap_insert()
// Same function handles BOTH max-heap and min-heap via is_max_heap flag

void heap_insert(Heap* h, const char* sym, double price, double chg, long vol) {
    int i = h->size++;

    // Bubble up — direction depends on heap type
    while(i > 0) {
        int parent = (i - 1) / 2;
        if(h->is_max_heap) {
            // MAX-HEAP: parent should be >= child
            if(h->data[parent].change < h->data[i].change)
                swap(&h->data[parent], &h->data[i]);
            else break;
        } else {
            // MIN-HEAP: parent should be <= child
            if(h->data[parent].change > h->data[i].change)
                swap(&h->data[parent], &h->data[i]);
            else break;
        }
        i = parent;
    }
}

Explain to faculty: "The is_max_heap flag controls the comparison direction. For the gainer heap, we want the biggest change_percent at the root (max-heap). For the loser heap, we want the most negative change at the root (min-heap). The same heapify logic works for both — only the comparison operator changes. This is an elegant code reuse pattern."

Top-K Extraction

// Extract root K times to get top-K gainers or losers
for(int i = 0; i < K && heap->size > 0; i++) {
    HeapStock top = heap_extract(heap);  // O(log n) per extraction
    printf("#%d: %s → %.2f%%\n", i+1, top.symbol, top.change);
}
// Total: O(K log n) for top-K elements

Section 08 — Member 5

Trie — Prefix Tree

What Is It? (Explain to Faculty)

A Trie (from retrieval, pronounced "try") is a tree-like data structure optimized for string operations. Each node represents a single character, and paths from root to leaf spell out stored keys. Unlike hash tables, tries support prefix queries — finding all words that start with a given prefix — in O(m + r) time, where m is the prefix length and r is the number of results.

In our project, the trie enables instant auto-complete: type "A" and immediately get AAPL, AMD, AMZN, ADBE, AVGO.

Where Is It Applied? (Code)

Trie Node — 26-ary Branching

// File: member5_trie.c
// Each node has 26 children — one for each letter A-Z

typedef struct TrieNode {
    struct TrieNode* children[26];  // 26-ary branching
    int    is_end;                   // 1 = complete word ends here
    double price;                    // Stock price at end node
    char   symbol[MAX_SYMBOL_LEN];   // Full symbol stored at end
} TrieNode;

Explain to faculty: "Each node has a 26-element pointer array — one slot per letter. To go from 'A' to 'P', we follow children[15] (P is the 16th letter, index 15). The is_end flag marks complete words. This 26-ary branching is the key design decision — it trades space for speed."

Auto-Complete — Prefix Search + DFS Collection

// File: member5_trie.c → get_suggestions()
// Step 1: Navigate to the prefix node
// Step 2: DFS to collect ALL complete words below that node

void get_suggestions(TrieNode* root, const char* prefix, ...) {
    TrieNode* curr = root;

    // Step 1: Walk to prefix node — O(m)
    for(int i = 0; prefix[i]; i++) {
        int idx = toupper(prefix[i]) - 'A';
        if(!curr->children[idx]) return;  // Prefix not in trie
        curr = curr->children[idx];
    }

    // Step 2: DFS collect all end-nodes — O(r)
    collect(curr, results, prices, count, max_results);
}

static void collect(TrieNode* node, ...) {
    if(node->is_end) {
        // Found a complete word — add to results
        strcpy(results[*count], node->symbol);
        (*count)++;
    }
    // Recurse through all 26 children
    for(int i = 0; i < 26; i++)
        if(node->children[i])
            collect(node->children[i], ...);
}

Explain to faculty: "Auto-complete works in two phases. Phase 1: Navigate to the prefix node by following each character — this is O(m) where m = prefix length. Phase 2: Run DFS from that node to collect all complete words below — this visits only relevant subtree nodes. For prefix 'A', we reach the 'A' node and then DFS finds AAPL, AMD, AMZN, ADBE, AVGO."

Member 1 — Hash Table (2 minutes)

"I implemented the Hash Table module for O(1) stock price lookup." [Action: Click "01 HASH TABLE" tab, then click "Load Demo"] "When I click 'Load Demo', all 12 stocks are inserted into a hash table with 100 buckets. Look at the console — it shows each stock being inserted with its hash index." "The hash function is djb2 — it multiplies by 33 and adds each character. For example, 'AAPL' hashes to index 11, and 'GOOGL' hashes to index 93." "Now look at this line: [COLLISION] at index 11 for AMZN. This means AMZN also hashed to index 11, same as AAPL. This is a collision. We handle it using separate chaining — both stocks are stored in a linked list at bucket 11." [Action: Point to the bucket visualization on the right — show bucket 11 with AAPL → AMZN chain] "The load factor is 12 stocks / 100 buckets = 0.12. This is very low, meaning few collisions. The max chain length is 2." "This maps to the Hashing section of our syllabus — hash functions, collision resolution, and load factor analysis."

Member 2 — Priority Queue (2 minutes)

"I implemented the Priority Queue using a Min-Heap for urgent stock alert processing." [Action: Click "02 PRIORITY QUEUE" tab, then click "Load Demo"] "In a stock market, not all alerts are equal. A CRITICAL alert like a trading halt must be processed before a routine price update. Our min-heap ensures this." "Watch the console as demo alerts are inserted in random priority order. Now when we extract, they come out in priority order — CRITICAL first, then HIGH, then MEDIUM, then LOW — regardless of insertion order." "The key operations are: — heapify_up: after inserting at the end, bubble the element up if it has higher urgency than its parent — heapify_down: after extracting the root, move the last element to the root and push it down Both operations are O(log n) because the heap is a complete binary tree with height log(n)." "This maps to Unit 2 — Single and Double Ended Priority Queues, which includes Leftist Trees and Binomial Heaps."

Member 3 — AVL Tree (2.5 minutes)

"I implemented the AVL Tree for storing historical stock prices." [Action: Click "03 AVL TREE" tab, then click "Load 20 Records"] "An AVL tree is a self-balancing BST. The balance factor at every node — height of left subtree minus height of right subtree — must be -1, 0, or +1." "I'm inserting 20 price records with timestamps in order. If this were a plain BST, we'd get a linked list of height 20 — O(n) search. But look at the AVL tree's height — it's only 4 or 5. That's O(log n)." "This happens because of automatic rotations. There are 4 cases: — LL (Left-Left heavy) → single right rotation — RR (Right-Right heavy) → single left rotation — LR (Left-Right) → double rotation: left-rotate child, then right-rotate node — RL (Right-Left) → double rotation: right-rotate child, then left-rotate node" "Each rotation is O(1) — just pointer reassignment. The console shows how many rotations occurred during the 20 insertions." "This maps to Unit 1 — AVL Trees, Red-Black Trees, B-Trees, and Splay Trees. AVL is the strictest balanced tree (BF ≤ 1). Red-Black trees allow BF up to 2 but do fewer rotations."

Member 4 — Heap (2 minutes)

"I implemented the dual Heap module — Max-Heap for top gainers and Min-Heap for top losers." [Action: Click "04 HEAP" tab, then click "Demo"] "When stocks are loaded, each stock is routed based on its change_percent: — Positive change → Max-Heap (gainers) — root holds the biggest gainer — Negative change → Min-Heap (losers) — root holds the biggest loser" "This is effectively a Double Ended Priority Queue (DEPQ) — we can extract the maximum from one heap and the minimum from another." "The elegant part is code reuse: the same heap_insert() function handles both modes using an is_max_heap boolean flag that controls the comparison direction." [Action: Click "Extract Gainer" to show the top gainer being removed, then "Extract Loser"] "Each extraction is O(log n). To get the top-5 gainers, we just extract 5 times." "This maps to Unit 1 (Heaps) and Unit 2 (DEPQs). The syllabus covers Min-Max Heaps and Deaps as alternatives — our dual-heap is a simpler but equally functional approach."

Member 5 — Trie (2 minutes)

"I implemented the Trie for stock symbol auto-complete." [Action: Click "05 TRIE" tab. If not loaded, click "Load Demo". To store 'AAPL', we create nodes A → A → P → L, marking L as an end-of-word." This is O(m + r) — m for reaching the prefix node, r for collecting results." [Action: Clear and type "NV"] "Now 'NV' gives just NVDA. The search narrows instantly as I type more characters." "The key advantage over hash tables: hash tables can do O(1) exact lookup, but they cannot do prefix queries. You can't ask a hash table 'give me all stocks starting with A'. That's what makes tries special." "The tradeoff: each node stores 26 pointers even if most are NULL. With 18 symbols we create 58 nodes using 11.8KB.
