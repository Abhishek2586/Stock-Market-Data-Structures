Section 01

Introduction — What AI Concepts We Use & Why

Yatra is a travel recommendation system that applies classical Artificial Intelligence (Symbolic AI) to solve a real-world problem: helping users find the best Indian travel destination based on their preferences. We implement 5 core AI concepts from the AI syllabus — without using any machine learning or neural networks.

Why These 5 AI Concepts?

#AI ConceptWhat It Does in Our ProjectWhy We Need It
1 Rule-Based Expert System Applies IF-THEN rules to decide if a destination fits the user's budget, season, and traveller type A travel agent follows rules — "If budget is low, skip luxury resorts." We encode this expert knowledge as rules so the system reasons like a human agent.
2 Fuzzy Logic Matches user interests approximately — "trek" finds "Trekking", "beech" finds "Beach" Users don't type perfectly. Fuzzy matching handles typos and partial inputs, just like how humans understand imprecise language.
3 Heuristic Scoring Assigns a weighted score (0–115) to each destination across 7 parameters We can't check every possible trip combination exhaustively. A heuristic function quickly estimates which destinations are "best" — this is how AI search works.
4 Knowledge Representation Stores 100 destinations with 53+ structured fields in JSON format AI needs organized knowledge to reason about. Our JSON dataset is the "brain" — it stores everything the system knows about Indian tourism.
5 Explainable AI (XAI) Shows WHY each destination is recommended — score, season badges, traveller reasons Unlike black-box ML, our system explains every decision. Users trust recommendations they can understand.
Important — What Makes This AI?

AI is not just machine learning. The AI syllabus covers Rule-Based Systems, Heuristic Search, Knowledge Representation, Expert Systems, and Fuzzy Logic — all of which we implement. These are the foundations of AI (Symbolic AI), predating neural networks by decades.

Architecture — How AI Components Connect

Three-layer architecture: Frontend → AI Engine → Knowledge Base, with all 5 AI concepts mapped

Layer 1: Frontend (HTML/CSS/JS) — The User Interface

5 pages where users interact with the AI system. The frontend collects preferences and displays AI-generated results with explanations.

Layer 2: AI Recommendation Engine (Python — recommender.py)

This is where all 5 AI concepts live. It contains:

  • Rule Engine — IF-THEN production rules for budget, season, traveller evaluation
  • Fuzzy Matcher — difflib-based approximate string matching for interests
  • Scoring Engine — 7-parameter weighted heuristic evaluation function
  • Explanation Generator — Produces human-readable reasons for each recommendation

Layer 3: Knowledge Base (JSON — india_tourism_dataset.json)

Knowledge Representation — 100 destinations × 53+ fields. This structured data is the system's "knowledge" that the AI engine reasons about.

How Data Flows During a Recommendation

Step 1: User enters preferences (budget, interest, month, traveller type)
           ↓
Step 2: Knowledge Base — Load all 100 destinations from JSON
           ↓
Step 3: Rule Engine — Apply budget rules, season rules, region constraints
         → Reject destinations that fail hard constraints (budget too low)
           ↓
Step 4: Fuzzy Matcher — Match user's interest text against trip_types
         → "trek" matches "Trekking" via substring + Levenshtein distance
           ↓
Step 5: Scoring Engine — Calculate weighted score for each surviving destination
         → Budget(30) + Interest(25) + Season(20) + Traveller(15) + Duration(10) + bonuses
           ↓
Step 6: Sort all destinations by score (descending) — Best-First Search
           ↓
Step 7: XAI — Attach explanations: why text, season badges, score display
           ↓
Step 8: Display ranked results with full transparency to user

Section 04 — Member 1

Rule-Based Expert System

What Is It? (Explain to Guide)

An Expert System is an AI program that mimics the decision-making ability of a human expert. It has three parts:

  • Knowledge Base — stores facts (our JSON dataset with 100 destinations)
  • Inference Engine — applies IF-THEN rules to derive conclusions (our score_place() function)
  • Explanation Facility — explains why a decision was made (our XAI output)

In our project, the system acts like a human travel agent. A real travel agent uses rules like: "If the client has a low budget, don't suggest 5-star resorts." We encode the same expertise as production rules.

Where Is It Applied? (Code)

Rule 1 — Budget Assessment

# File: recommender.py → score_place()
# This production rule evaluates budget fit

if budget_low <= user_budget <= budget_high:
    score += 30        # RULE: Budget fits → full points
elif mid_low <= user_budget <= mid_high:
    score += 15        # RULE: Mid-range fit → partial points
elif user_budget < budget_low:
    return None        # RULE: Can't afford → HARD REJECT
else:
    score += 20        # RULE: Above mid-range → good fit

Explain to guide: "This is a production rule — an IF-THEN structure. The condition checks budget ranges, and the action assigns score points or rejects the destination. This is exactly how production systems work in Unit 1 of our syllabus."

Section 05 — Member 2

Fuzzy Logic & Fuzzy Matching

What Is It? (Explain to Guide)

Fuzzy Logic deals with reasoning that is approximate rather than exact. In classical (crisp) logic, something is either True or False. In fuzzy logic, there are degrees of truth — something can be "partially true" or "almost matching".

In our project, when a user types "trek", classical matching would say it does NOT equal "Trekking" (False). But fuzzy matching says it's close enough (almost True) — and finds the correct match. This is how humans think: we don't need exact words to understand meaning.

Where Is It Applied? (Code)

Python Backend — difflib.get_close_matches()

import difflib

def fuzzy_match_interest(user_input, trip_types, cutoff=0.6):
    """Match user's interest against destination types using 3 methods."""
    user_lower = user_input.lower()
    
    for trip_type in trip_types:
        type_lower = trip_type.lower().replace('_', ' ')
        
        # METHOD 1: Substring check — "trek" is inside "trekking"
        if user_lower in type_lower or type_lower in user_lower:
            return True
        
        # METHOD 2: Edit distance — how many edits to transform one into other
        if levenshtein_distance(user_lower, type_lower) <= 2:
            return True
        
        # METHOD 3: Similarity ratio via difflib (0.0 to 1.0)
        matches = difflib.get_close_matches(user_lower, [type_lower], cutoff=cutoff)
        if matches:
            return True
    
    return False

JavaScript Frontend — Levenshtein Distance

// File: index.html — Levenshtein edit distance algorithm
function lev(a, b) {
    // Dynamic programming approach to compute minimum edit distance
    const m = a.length, n = b.length, d = [];
    for (let i = 0; i <= m; i++) d[i] = [i];
    for (let j = 0; j <= n; j++) d[0][j] = j;
    for (let i = 1; i <= m; i++)
        for (let j = 1; j <= n; j++)
            d[i][j] = a[i-1] === b[j-1]
                ? d[i-1][j-1]               // No edit needed
                : 1 + Math.min(
                    d[i-1][j],              // Delete
                    d[i][j-1],              // Insert
                    d[i-1][j-1]             // Replace
                );
    return d[m][n]; // Minimum edits needed
}

"In our project, we implement Fuzzy Logic for handling user inputs. When a user types their travel interest, we don't require an exact match. This means if someone types 'trek', our system understands they mean 'Trekking'. The difflib library computes a similarity score from 0 to 1, which is analogous to a fuzzy membership function — the higher the score, the closer the match. We set a threshold of 0.6, which is our alpha-cut level."

Heuristic AI Scoring System

What Is It? (Explain to Guide)

A heuristic is a problem-solving technique that uses practical methods to find a "good enough" answer quickly, rather than searching exhaustively. In AI, a heuristic evaluation function h(n) estimates how good a particular solution is.

In our project, we can't try every possible combination of user preferences and destinations. Instead, we use a weighted scoring function that quickly evaluates each destination's suitability — this is our heuristic. We then sort by score (highest first), which implements Best-First Search.

The 7 Scoring Parameters

#ParameterMax PointsWeight ReasonHow It Works
1Budget Fit30Highest — money is the primary constraint+30 (budget tier), +20 (luxury), +15 (mid), reject (too low)
2Interest Match25High — must match what user wants to do+25 (fuzzy match), reject (no match found)
3Season20High — wrong season ruins a trip+20 (best), +8 (neutral), -15 (avoid)
4Traveller Type15Medium — comfort depends on group type+15 (ideal for this type), +3 (not ideal)
5Duration10Medium — trip length must fit+10 (within range), -5 (too short)
6Popularity10Bonus — popular places are generally goodAdds popularity_score directly (1-10)
7Safety5Bonus — safety matters for peace of mindAdds safety_rating ÷ 2 (1-5)

Maximum possible score: ~115 points (30+25+20+15+10+10+5)

Where Is It Applied? (Code)

# File: recommender.py → score_place()
# This is the HEURISTIC EVALUATION FUNCTION h(n)
# h(n) = w₁·budget + w₂·interest + w₃·season + w₄·traveller + w₅·duration + popularity + safety/2

def score_place(place, budget, interests, season, traveller, days):
    score = 0
    
    # Each parameter adds/subtracts from total score
    score += budget_score(place, budget)       # max +30
    score += interest_score(place, interests)   # max +25
    score += season_score(place, season)        # max +20
    score += traveller_score(place, traveller)  # max +15
    score += duration_score(place, days)        # max +10
    score += place.popularity_score             # max +10
    score += place.safety_rating // 2          # max +5
    
    return score  # Total heuristic value

# Then in recommend():
results = []
for place in all_destinations:
    s = score_place(place, ...)
    if s is not None:
        results.append((place, s))

# BEST-FIRST SEARCH: Sort by score descending
results.sort(key=lambda x: x[1], reverse=True)
return results[:top_n]  # Return top N best matches

Section 07 — Member 4

Knowledge Representation

What Is It? (Explain to Guide)

Knowledge Representation (KR) is how an AI system stores information about the world so it can reason about it. Without organized knowledge, an AI system has nothing to reason with — it's like a brain without memories.

Our project uses a structured JSON knowledge base — a symbolic representation where each destination is described by 53+ named fields organized into categories. This is analogous to frame-based representation in AI, where each frame (destination) has slots (fields) with values.

Explainable AI (XAI)

What Is It? (Explain to Guide)

Explainable AI (XAI) means AI systems whose actions and decisions can be understood by humans. When a neural network says "buy this stock," you don't know WHY. But when our expert system says "visit Goa," it explains: "Because it fits your budget, Beach matches your interest, Winter is the best season, and it's ideal for couples."

XAI builds trust — users trust recommendations they can understand. In our Expert System, the Explanation Facility is one of the three required components (alongside KB and Inference Engine).

# File: recommender.py → print_card()

def print_card(place, score, traveller_type, season):
    # XAI Feature 1: Transparent Score
    print(f"SCORE: {score}/~115")
    
    # XAI Feature 2: Season Explanation

        print("✓ BEST TIME — This is the ideal season to visit")
    elif season in place.avoid_seasons:

    # XAI Feature 3: Personalized "Why" Explanation
    if traveller_type in place.ideal_for_why:
        print(f"Why this suits you: {place.ideal_for_why[traveller_type]}")
    
    # XAI Feature 6: Permit Warning
    if place.permits_required:
        print(f"⚠ PERMIT REQUIRED: {place.permit_details}")

Member 1 — Rule-Based Expert System (2 minutes)

"Now I'll explain the Rule-Based Expert System." "An Expert System has three components: a Knowledge Base — which we just saw, an Inference Engine — which applies rules, and an Explanation Facility — which explains decisions." "Our inference engine uses IF-THEN production rules. Let me demonstrate." [Action: Go to Find Trip page] "When I enter a budget of ₹3,000 per day, the system applies Rule 1: Budget Assessment. — IF the destination's budget range includes ₹3,000 → add 30 points — IF it's only in mid-range → add 15 points — IF ₹3,000 is below the minimum → HARD REJECT — skip this destination entirely This is a Constraint Satisfaction problem from Unit 3." "When I select January as the travel month, Rule 2: Season Detection fires. — The system maps January → Winter using a rule table — IF Winter is in the destination's best_seasons → +20 points bonus — IF Winter is in avoid_seasons → -15 points penalty" "When I select Couple as traveller type, Rule 3: Traveller Match checks: — IF Couple is in the destination's ideal_for list → +15 points — ELSE → only +3 points" "This is exactly the Production System from Unit 1 and the Expert System Design from Unit 5."

Member 2 — Fuzzy Logic (1.5 minutes)

"Let me explain the Fuzzy Logic component." "When a user selects or types their interest — say 'Beach' — we need to match this against each destination's trip_types. But what if someone types 'beech' with a typo, or just 'trek' instead of 'Trekking'?" "This is where Fuzzy Matching comes in. Instead of requiring exact matches, we use three methods: 1. Substring matching — 'trek' is found inside 'trekking' 2. Levenshtein edit distance — 'beech' needs only 1 edit to become 'beach' (≤ 2 threshold) 3. difflib similarity ratio — computes a score from 0.0 to 1.0, we accept anything above 0.6" "In our Python backend, we use difflib.get_close_matches() from Python's standard library. In the JavaScript frontend, we implemented the Levenshtein distance algorithm using dynamic programming." "This relates to Fuzzy Logic Systems in the AI syllabus. The similarity ratio is analogous to a fuzzy membership function — it gives a degree of truth (how similar the strings are) rather than just True or False." [Action: Point to the interest chips on Find Trip page — show how Beach matches results with Beach in their trip_types]
